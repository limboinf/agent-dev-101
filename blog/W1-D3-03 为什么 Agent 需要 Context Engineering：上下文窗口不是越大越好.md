# 为什么 Agent 需要 Context Engineering：上下文窗口不是越大越好

---

> 摘要：上一篇我们搞懂了 Context Engineering 的定义和四大支柱，你可能会想——"既然上下文窗口越来越大（Claude 200K、Gemini 1M），那直接把所有信息都塞进去不就行了？"这篇我们来打碎这个幻觉。从 "Lost in the Middle" 到 Context Rot，从模型注意力的 U 型分布到 Agent 多步循环的特殊挑战——你会理解为什么 Context Engineering 不是可选项，而是 Agent 开发的核心技能。

## 一、一个直觉：为什么聊天越长，AI 越"笨"？

用过 ChatGPT 或 Claude 的人大概都有这个体验：

- 刚开始对话时，模型回答得又快又准
- 聊了十几轮之后，它开始忽略你之前说过的话
- 聊了三十轮以后，它甚至开始重复自己，或者"忘记"你早就说过的约束条件

你可能以为这是 bug，或者模型在偷懒。但实际上，**这是 LLM 的根本特性**——它和你聊得越久，消息列表越长，它能"记住"每条消息的能力就越弱。

上一篇我们提到的那个客服 Agent 例子就是典型：30 轮对话 + 工具结果 + RAG 检索结果 = 15000 tokens。模型不是"忘了"退货单号，而是在 15000 个 token 的信息海洋里，**它没有足够的注意力分配给那个关键信息**。

这种现象有一个学术名字：**Context Rot（上下文腐化）**。

---

## 二、"Lost in the Middle"：模型的注意力是 U 型的

2023 年，Nelson Liu 等人发表了一篇影响深远的论文 "Lost in the Middle"。他们用 "Needle-in-a-Haystack" 任务（在大量文本中找到一句特定的话）测试了多个 LLM，发现了一个令人不安的规律：

**模型对输入开头和结尾的信息关注度最高，而中间的信息最容易被忽略。**

```text
注意力强度（示意）:

高 ▕████                                    ████▏
   ▕███                                      ███▏
   ▕██                                        ██▏
   ▕█              ........                    █▏
低 ▕                ........                     ▏
   └───────────────────────────────────────────┘
   开头            中间                      结尾
```

上一篇消息管理我们提到过这个问题，这里深入展开。

### 更细致的发现

后续的研究让我们对这个问题的理解更加精细：

**1. Veseli et al (2025) 发现：注意力模式随上下文填充度变化**

| 上下文填充度 | 注意力模式 | 具体表现 |
|-------------|-----------|---------|
| < 50% | U 型分布 | 开头和结尾的信息关注度高，中间被忽略 |
| > 50% | 近因偏差 | 越靠近结尾的信息关注度越高，早期信息几乎被遗忘 |

这意味着：当你的上下文窗口只用了一小半时，System Prompt（开头）和最新消息（结尾）都能被模型"看到"；但当上下文窗口用了大半，**早期的指令和约束就开始被忽略了**。

这就解释了为什么 Claude Code 在上下文满了以后，有时候会无视你 `CLAUDE.md` 里写的规则——因为那些规则在上下文的最前面，而当窗口塞满以后，模型的注意力偏向了最近的信息。

**2. Du et al (2025) 的关键发现：这不是"找不到"，而是"处理不了"**

很多人的直觉是：模型在长上下文中表现差，是因为它"检索"不到关键信息——就像你在一堆文件里找不到那张发票。

但 Du et al 做了一个巧妙的实验：他们把所有的"干草"（非关键信息）都替换成了空格，只留下"针"（关键信息）。按照"检索困难"的假说，这时候"针"应该很容易被找到才对。

**结果：模型的表现并没有显著改善。**

这说明问题不在于"找不到"，而在于**输入长度本身就会降低模型的处理能力**。这是一个更根本的限制——不是 UX 问题，是模型架构的固有约束。

## 三、Context Rot：上下文窗口越大，不代表越好

### 什么是 Context Rot？

Chroma Research 的研究给这个现象起了一个生动的名字：**Context Rot（上下文腐化）**。

> 随着上下文窗口中 Token 数量的增加，模型准确回忆信息的能力会下降。

Anthropic 在他们的 Context Engineering 文章中也验证了这一点：

> "We've observed that LLMs, like humans, lose focus or experience confusion at a certain point."
> （我们观察到，LLM 和人类一样，到了某个点就会失去注意力或产生混乱。）

### 四种 Context 失败模式

Drew Breunig 把 Context Rot 的具体表现总结为四种失败模式：

| 失败模式 | 含义 | Agent 中的典型场景 |
|---------|------|------------------|
| **Context Poisoning**（上下文中毒） | 幻觉产生的错误信息混入上下文，后续步骤基于错误信息继续推理 | Agent 第 3 步产生了一个错误的中间结果，第 5 步基于这个错误结果做决策 |
| **Context Distraction**（上下文干扰） | 上下文中的信息压过了模型训练时学到的知识 | 塞了太多 RAG 检索结果，模型反而不敢用自己的知识回答简单问题 |
| **Context Confusion**（上下文混淆） | 多余的、不相关的信息干扰模型的响应 | 注册了 20 个工具，模型在选择工具时犹豫不决或选错 |
| **Context Clash**（上下文冲突） | 上下文中的不同部分互相矛盾 | System Prompt 说"简洁回答"，但 Few-shot 示例全是长回答 |

### 1M Token 窗口不是银弹

看到这里你可能会想：那既然窗口大小是问题，用更大的窗口不就行了？

2026 年，我们已经有了 Claude 的 200K（甚至 1M）token 窗口、GPT 的 400K、Gemini 的 1M。但研究一再证明：

**更大的窗口解决了"装不下"的问题，但没有解决"注意力不够"的问题。**

一个类比：把你的书桌从 1 米宽换成 3 米宽，你确实能放更多文件了。但你的眼睛还是那双眼睛——你不可能同时看清桌上的每一张纸。桌子越大，反而越容易找不到你需要的那张。

```text
小窗口的问题：装不下（信息丢失）
大窗口的问题：信号被稀释（信息淹没）

     小窗口          大窗口
   ┌─────────┐    ┌───────────────────────────┐
   │ 重要  重要 │    │ 重要  噪音  噪音  噪音  重要 │
   │ 信息  信息 │    │ 噪音  噪音  重要  噪音  噪音 │
   └─────────┘    │ 噪音  噪音  噪音  噪音  信息 │
    信号密度高       └───────────────────────────┘
                     信号密度低，关键信息被稀释
```

所以 Context Engineering 的目标不是"用更大的窗口"，而是**在任何大小的窗口里，最大化信号密度**。

## 四、Agent 为什么特别需要 Context Engineering

到这里你可能会说：好吧，LLM 的注意力确实有限。但为什么 Agent 比普通聊天场景更需要 Context Engineering？

答案在于 Agent 的核心运作模式：**多步循环**。

### Agent 的上下文膨胀问题

一个普通的聊天对话，上下文增长是线性的：每轮加一条用户消息和一条助手回复。

但 Agent 不一样。一个典型的 Agent 循环长这样：

```text
Step 1: 用户提问                         → +100 tokens
Step 2: 模型推理 + 选择工具                → +200 tokens
Step 3: 工具调用结果（搜索结果）           → +2000 tokens
Step 4: 模型基于结果继续推理               → +300 tokens
Step 5: 再次调用工具（数据库查询）         → +1500 tokens
Step 6: 模型整合结果                       → +400 tokens
Step 7: 又一次工具调用（代码执行）         → +1000 tokens
Step 8: 最终回答                           → +500 tokens
─────────────────────────────────────────────────
总计:                                       ~6000 tokens（单次任务）
```

**一次任务就消耗 6000 tokens**。如果 Agent 在一个会话中连续处理 10 个任务，上下文就膨胀到 60000+ tokens。

更糟的是，工具调用的结果往往是"重"的——一次搜索返回 2000 tokens，一次文件读取可能 5000 tokens。这些"重量级"内容会快速填满上下文窗口。

Anthropic 在他们的多 Agent 研究系统中观察到：

> "Agents often engage in conversations spanning hundreds of turns, requiring careful context management strategies."
> （Agent 经常进行跨越数百轮的对话，需要仔细的上下文管理策略。）

### Agent 的每一步需要不同的"最优上下文"

这是 Agent 和普通对话最大的区别。在聊天场景中，每一轮需要的上下文大致相同：历史对话 + 当前问题。

但 Agent 的每一步需要的信息是不同的：

| Agent 步骤 | 需要的上下文 | 不需要的上下文 |
|-----------|------------|-------------|
| **规划阶段** | 全局目标、当前状态、可用工具列表 | 之前步骤的原始工具输出 |
| **工具选择** | 工具定义（JSON Schema）、用户意图 | 早期对话历史 |
| **结果处理** | 工具返回值、原始用户问题 | 其他工具的定义 |
| **最终回答** | 用户问题、关键中间结果 | 所有工具调用的原始请求/响应 |

如果你不做任何管理，每一步都带上所有历史信息，那就是在浪费模型宝贵的注意力——让它在一堆它这一步根本不需要的信息里"找"它需要的东西。

### Chatbot vs Agent：上下文复杂度对比

```text
Chatbot 的上下文（简单、线性）:
┌────────────────────────────────────────┐
│ System Prompt                          │
│ User: 你好                              │
│ Assistant: 你好！有什么可以帮你的？          │
│ User: 帮我写一首诗                        │
│ Assistant: 好的，这是一首关于春天的诗...      │
└────────────────────────────────────────┘

Agent 的上下文（复杂、多源、动态）:
┌────────────────────────────────────────┐
│ System Prompt（角色 + 约束 + 输出格式）    │
│ 工具定义 ×5（JSON Schema，~2000 tokens） │
│ User: 帮我查一下北京明天天气，如果超过30度...  │
│ Assistant: [tool_call: get_weather]    │
│ Tool Result: {"temp": 35, ...}         │
│ Assistant: [tool_call: recommend_food] │
│ Tool Result: {"items": [...]}          │
│ RAG 检索结果: [冷饮推荐指南...]             │
│ Assistant: 北京明天35度，推荐以下冷饮...     │
│ ... （更多轮次，上下文持续膨胀）             │
└────────────────────────────────────────┘
```

Agent 的上下文不仅更长，而且**更杂**——混合了指令、工具定义、工具结果、检索结果、对话历史、推理痕迹……每一种信息的"信号强度"和"时效性"都不同，管理难度远超普通对话。

## 五、核心目标：最小、最高信号的 Token 集合

理解了上面的问题之后，Context Engineering 的核心目标就清晰了。

Anthropic 在他们的 Context Engineering 文章中总结得很精辟：

> "Find the smallest set of high-signal tokens that maximize the likelihood of your desired outcome."
> （找到最小的、信号最强的 Token 集合，最大化你期望结果的可能性。）

不是"给模型尽可能多的信息"，而是**"给模型恰好需要的信息"**。

### 来自真实产品的佐证

**1. Vercel 的教训：删掉 80% 的工具，效果反而更好**

Vercel 在开发他们的 AI 助手时发现，注册了太多工具后，模型在选择工具时反而犯更多错误。他们尝试删掉了 80% 的工具——**效果显著提升**。

这就是 Context Confusion 的实际案例：模型不是不够聪明，而是你给了它太多选择。工具定义本身就是上下文的一部分，每个工具的 JSON Schema 都要消耗几百个 Token。20 个工具可能就占了 4000-5000 tokens 的上下文空间，但这一步可能只需要 2 个工具。

**2. Claude Code 的 auto-compact：95% 触发压缩**

Claude Code 在上下文窗口使用率达到 95% 时，会自动触发 "auto-compact"——用 LLM 对整个对话历史做一次摘要压缩。压缩时保留关键决策、未解决的问题和实现细节，丢弃冗余的工具输出。

```text
Claude Code 的 auto-compact 策略：

上下文使用率:
0%           50%          95%    100%
|─────────────|──────────────|──|
                              ↑ 
                         触发 auto-compact
                         
压缩后：
- 保留：架构决策、未解决的 bug、关键代码变更
- 丢弃：旧的工具调用原始结果、冗余的中间推理
- 附加：最近 5 个访问过的文件内容
```

**3. Manus 的 Pre-Rot Threshold：提前到 70% 就开始干预**

Manus 比 Claude Code 更激进——他们在上下文使用率达到 **70%** 时就开始压缩，而不是等到 95%。他们把这个阈值叫做 "Pre-Rot Threshold"（预腐化阈值）。

为什么？因为研究表明，上下文填充度超过 50% 后，模型的注意力模式就从 U 型变成了纯粹的近因偏差——**早期信息已经开始被忽略了**。等到 95% 才压缩，你可能已经丢失了好几步的关键决策信息。

### 质量 > 数量

这三个案例传达了同一个信息：**上下文的质量远比数量重要**。

```text
❌ 错误的思维模型：
   "窗口有 200K tokens，我尽量塞满它"

✅ 正确的思维模型：
   "这一步模型需要什么信息？只给它这些"
```

## 六、现阶段你应该建立的心智模型

学到这里，关于 Context Engineering 的"为什么"我们已经讲清楚了。在继续学习具体的模式和技巧之前，先把几个核心认知锚定下来：

### 1. 上下文窗口 = 稀缺资源

不管窗口有多大——128K、200K、1M——它都是有限的。而且随着填充度增加，模型的表现会下降。把上下文窗口当成你最宝贵的资源来管理。

### 2. 每条消息都有成本

不仅是金钱成本（Token 计费），还有"注意力成本"——每多一条无关消息，模型分配给关键信息的注意力就少一点。

### 3. 信号密度比信息量更重要

10 条高信号的消息 > 100 条混杂着噪音的消息。Context Engineering 的核心就是**提高信号密度**。

### 4. Agent 的每一步都应该精心策划

不是所有步骤都需要看到所有信息。规划步骤看全局，执行步骤看细节，总结步骤只看结果——**按需加载，用完就清**。

### 5. "给得少但给得对"胜过"给得多但给得乱"

这是 Context Engineering 的核心哲学。Anthropic 把它概括为一句话："Find the smallest set of high-signal tokens."

```text
Context Engineering 的核心使命：

  ┌──────────────────────────────────────┐
  │  在 Agent 的每一个推理步骤            │
  │  构造最小的、信号最强的上下文           │
  │  使模型做出最优的下一步决策             │
  └──────────────────────────────────────┘
```

## 小结

这篇我们深入探讨了 Agent 为什么特别需要 Context Engineering：

1. **"Lost in the Middle"**：模型注意力呈 U 型分布，中间的信息最容易被忽略；填充度超过 50% 后变成纯粹的近因偏差
2. **Context Rot**：上下文越大≠越好，四种失败模式（Poisoning、Distraction、Confusion、Clash）随时可能发生
3. **Agent 的特殊挑战**：多步循环导致上下文快速膨胀，且每一步需要不同的"最优上下文"
4. **核心目标**：不是"塞满窗口"，而是"在每一步提供最小、最高信号的 Token 集合"
5. **真实案例**：Vercel 删工具、Claude Code auto-compact、Manus Pre-Rot Threshold——都在践行同一个原则

关键心智模型：**上下文窗口不是"越大越好"的仓库，而是"越精越好"的工作台。你放上去的每一份信息都应该是这一步真正需要的。**

> 💡 **预告**：理解了"为什么需要 Context Engineering"之后，下一篇我们进入实操——学习 Context Engineering 的四大核心模式：渐进披露（Progressive Disclosure）、上下文压缩（Compression）、上下文路由（Routing）和工具管理（Tool Management）。这些是你在 Agent 开发中每天都会用到的实战技巧。
