# Prompt Engineering 回顾：从会写提示词到理解"信息设计"

---

> 摘要：上一篇我们搞定了消息管理——怎么数 Token、怎么截断、怎么压缩。但管好了"容器"之后，容器里到底该"装什么"？这篇我们回到最基本的功夫：Prompt Engineering。Zero-shot、Few-shot、CoT 这些经典技巧你可能都听过，但在 Agent 开发的语境下重新审视它们，你会发现 Prompt Engineering 并不是"过时的技术"——它是 Context Engineering 的起点和子集。

## 一、为什么要在这个时候回顾 Prompt Engineering？

上一篇讲消息管理的时候，我们一直在纠结一个问题：**消息列表太长了怎么办？** 截断、压缩、摘要……全是在处理"量"的问题。

但你有没有想过：就算你把消息列表管理得很好，**消息本身写得烂，模型输出照样拉跨**。

这就是 Prompt Engineering 要解决的问题：**怎么把你的意图准确、完整地传递给模型**。

而且有一个容易被忽略的事实：在 Agent 系统中，**每一轮循环的 System Prompt、每一次工具调用的描述、每一条注入的上下文**——这些全都是 Prompt。你以为 Prompt Engineering 只是"写好一句话"，实际上它是 Context Engineering 最底层的砖块。

---

## 二、三种经典 Prompt 技巧：Zero-Shot / Few-Shot / Chain-of-Thought

这三个是 Prompt Engineering 的"三板斧"，几乎所有场景都绕不开它们。

### 1. Zero-Shot：直接给指令，不给示例

Zero-shot 的核心假设是：**模型已经具备完成这个任务的能力，你只需要把任务描述清楚**。
但问题是——"描述清楚"本身就很难。你说"分类"，模型可能返回一段解释性文字，也可能只返回一个标签。这就是为什么光靠 Zero-shot 经常翻车。

### 2. Few-Shot：给几个示例，让模型"学会"

如果你给模型看几个"输入→输出"的例子，它就能非常准确地模仿这个模式：

```python
response = client.chat.completions.create(
    model="qwen3.5-flash",
    messages=[
        {
            "role": "user",
            "content": """将用户反馈分类为 '功能请求'、'Bug 报告' 或 '使用咨询'。

示例：
反馈：希望能支持暗黑模式 → 功能请求
反馈：上传文件时报 500 错误 → Bug 报告
反馈：怎么修改个人头像？ → 使用咨询

现在请分类：
反馈：能不能增加批量导出功能"""
        }
    ]
)
print(response.choices[0].message.content)
# 输出: 功能请求
```

Few-shot 的关键是：**示例不是在"教"模型新知识，而是在告诉它你期望的输出格式和判断标准**。

一般 2-5 个示例就够了。示例太多反而会浪费 Token，而且可能引入偏差。

### 3. Chain-of-Thought（CoT）：让模型一步步推理

CoT 的核心发现是：**在复杂任务中，让模型先"想"再"答"，准确率会显著提高**。

```python
...
请一步一步思考，然后给出最终答案 ...
```

加上"请一步一步思考"这句话，模型会先输出推理过程然后再给出答案。推理过程的存在让最终答案更准确。

> 有意思的是，现在的推理模型其实是在模型层面内置了 CoT——它们会自动进行"内部推理"。但在使用非推理模型时，**手动触发 CoT 仍然是最有效的提升手段之一**。

### 三种技巧的选择

| 技巧 | 适用场景 | Token 成本 | 典型效果 |
|------|---------|-----------|---------|
| Zero-shot | 简单、明确的任务（翻译、摘要、分类） | 最低 | 对强模型效果好，弱模型容易偏 |
| Few-shot | 需要特定格式/判断标准的任务 | 中等（示例占 Token） | 格式一致性高，判断标准更稳定 |
| CoT | 多步推理、数学、逻辑判断 | 较高（推理过程占 Token） | 复杂任务准确率显著提升 |
我的经验是：**先用 Zero-shot 试一下，不行就加 Few-shot，还不行就上 CoT**。别一上来就写一大堆示例和推理链——那些 Token 都是钱。

## 三、System Prompt 设计：角色设定、约束、输出格式

如果说 Zero-shot / Few-shot / CoT 是"怎么问"的技巧，那 System Prompt 就是**在对话开始前，给模型搭好一个"工作环境"**。

一个好的 System Prompt 通常包含三个核心要素：

### 1. 角色设定（Role）

告诉模型"你是谁"：

角色设定不是玄学。它的作用是**激活模型在训练数据中学到的特定领域知识和行为模式**。当你说"你是一个资深 Python 专家"，模型会倾向于使用更专业的术语、更严谨的判断标准。

但也不要过度角色扮演：

```python
# ❌ 过度角色扮演
system_prompt = """你是一个叫Limbo的程序员，在北京中关村上班，
喜欢喝咖啡，养了一只猫叫xx……"""

# ✅ 聚焦于任务相关的角色特征
system_prompt = """你是一个 Python 代码审查专家，专注于：
- 代码可读性和命名规范
- 性能优化建议
- 安全漏洞检测"""
```

### 2. 约束条件（Constraints）

告诉模型"什么能做，什么不能做"：

约束条件是 Agent 开发中**最重要也最容易被忽略**的部分。没有约束的 Agent 就像没有护栏的马路——迟早出事。

### 3. 输出格式（Output Format）

告诉模型你期望的输出长什么样：

```python
system_prompt = """你是一个代码审查助手。

对于每个代码片段，请按以下格式输出审查结果：

## 总体评价
[一句话总结代码质量]

## 问题列表
1. [严重程度: 高/中/低] [问题描述]
   - 位置：[行号或代码片段]
   - 建议：[具体的修改建议]

## 改进后的代码
```改进后的完整代码```"""
```

指定输出格式能极大提升模型输出的一致性和可解析性。在 Agent 系统中，这一点尤为重要——因为你的代码需要程序化地处理模型的输出。

一个成熟的 System Prompt 通常包含以下模块：

```
# [角色定义]
你是一位...（身份 + 专业背景）

# [核心目标]
你的主要任务是...

# [行为规则]
- 必须做的事
- 禁止做的事

# [输出格式]
回复格式为...（结构/长度/语言/风格）

# [边界与约束]
当遇到...时，你应该...
```

OpenAI 官方提炼的六大核心策略至今仍是业界基准 ：
1. **写清晰的指令** — 避免模糊，指定输出长度、格式、语言、受众
2. **提供参考文本** — 给出示例输出（Few-shot），让模型对齐预期
3. **拆解复杂任务** — 将多步骤任务分解，避免单一 Prompt 过载
4. **给模型思考时间** — 使用 Chain-of-Thought，引导逐步推理（"先分析…，再得出结论…"）
5. **结合外部工具** — 在需要实时数据时，明确告知模型调用工具
6. **系统性测试迭代** — 准备 10-20 个覆盖典型和边界情况的测试集，验证 Prompt 效果

## 四、Prompt Engineering 在 Agent 中的位置

讲了这么多 Prompt 技巧，是时候把它放回 Agent 开发的大背景下了。

### PE 在单轮静态任务中非常有效

如果你的需求是：
- 给一段文本做分类
- 翻译一篇文章
- 从一段话中提取结构化信息
- 回答一个知识性问题

那 Prompt Engineering 就够了。精心设计好 System Prompt + 选对技巧（Zero-shot / Few-shot / CoT），能解决 80% 的单轮任务。

### 但 Agent 是多轮、动态、工具交互的

Agent 的典型执行流程是这样的：

```text
用户请求 → 模型推理 → 调工具A → 拿结果 → 模型再推理 → 调工具B → ……→ 最终回答
```

在这个过程中，每一步模型看到的上下文都不一样：

```text
第 1 步: [System Prompt] + [用户请求]
第 2 步: [System Prompt] + [用户请求] + [模型推理] + [工具 A 结果]
第 3 步: [System Prompt] + [用户请求] + [模型推理] + [工具 A 结果] + [模型推理] + [工具 B 结果]
```

问题来了：

1. **上下文在增长**：每一步都比上一步多出新的信息
2. **信息有时效性**：第 1 步的工具结果在第 5 步可能已经不重要了
3. **不同步骤需要不同的信息**：调用搜索工具时需要看到搜索相关的上下文，调用计算工具时需要看到数字和公式等
4. **工具定义本身就占 Token**：10 个工具的 JSON Schema 可能就占了 2000-3000 Token

**Prompt Engineering 解决了"怎么问"，但它无法解决"模型在每一步应该看到什么信息"这个问题。**

这就是为什么我们需要从 Prompt Engineering 升级到 Context Engineering：

| 维度   | Prompt Engineering | Context Engineering   |
| ---- | ------------------ | --------------------- |
| 关注点  | 一条消息怎么写            | 模型在每一步看到的全部信息         |
| 适用范围 | 单轮、静态任务            | 多轮、动态、工具交互            |
| 核心问题 | 怎么问                | 模型看到什么                |
| 设计对象 | 一个 Prompt          | 整个信息流动系统              |
| 复杂度  | 中等                 | 高（需要考虑信息的选择、排序、压缩、时机） |

Andrej Karpathy 有一个经典比喻：**LLM 是 CPU，上下文窗口是 RAM，而你是负责加载正确信息的操作系统**。Prompt Engineering 只是在写"一条指令"，Context Engineering 是在设计"整个操作系统的内存管理策略"。

---

## 小结

这篇我们回顾了 Prompt Engineering 的核心技巧，关键收获：

1. **三种基础技巧**：Zero-shot（直接指令）→ Few-shot（给示例）→ CoT（逐步推理），先简单后复杂，别过度设计
2. **System Prompt 三要素**：角色设定 + 约束条件 + 输出格式，缺一不可
3. **模型差异**：Claude 偏爱 XML 标签结构化，OpenAI 用 Markdown 就行，针对不同模型调整格式
4. **PE 的边界**：在单轮静态任务中很有效，但 Agent 的多轮动态场景需要从"怎么问"升级到"模型看到什么"
