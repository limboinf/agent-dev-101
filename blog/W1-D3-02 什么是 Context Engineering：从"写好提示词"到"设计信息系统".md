# 什么是 Context Engineering：从"写好提示词"到"设计信息系统"

---

> 摘要：上一篇我们回顾了 Prompt Engineering 的经典技巧，你可能觉得"会写好 Prompt 就够了"。但当你开始做 Agent——消息列表越来越长、工具越来越多、检索结果塞了一堆——你会发现，真正的瓶颈不是"怎么问"，而是"模型到底看到了什么"。这就是 Context Engineering 要解决的问题。本篇带你搞懂它的定义、它和 Prompt Engineering 的核心区别、它的四大支柱，以及 LangChain 总结的四种实战策略。

![](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/20260413150942813.png)

## 一、Context Engineering 的定义

2025 年中，前 OpenAI 联合创始人 Andrej Karpathy 给了一个非常形象的比喻：

> "Think of the LLM as the CPU, the context window as RAM, and you are the operating system responsible for loading the right information at the right time."
>
> ——把 LLM 想象成 CPU，上下文窗口是内存（RAM），而你是操作系统，负责在正确的时间加载正确的信息。

他进一步把 Context Engineering 定义为：

> **"The delicate art and science of filling the context window with just the right information for the next step."**
>
> ——用恰到好处的信息填满上下文窗口的精妙艺术与科学。

这个比喻精准地揭示了一件事：**你不是在"写提示词"，你是在做内存管理。**

操作系统不会把磁盘上所有数据都塞进 RAM——它只加载当前进程需要的页面。同理，你不应该把所有信息都塞进上下文窗口——你应该只加载当前步骤需要的信息。

![](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/20260413145135009.png)

### 一句话总结

> **Context Engineering = 设计和管理 LLM 在每一步看到的全部输入信息的架构。**

## 二、Context Engineering vs Prompt Engineering

很多人第一次听到 Context Engineering 会想：这不就是 Prompt Engineering 换了个说法吗？ 不是。区别很大。

### 对比表格

| 维度       | Prompt Engineering | Context Engineering                  |
| -------- | ------------------ | ------------------------------------ |
| **关注点**  | 怎么问                | 模型看到什么信息                             |
| **作用范围** | 单次                 | 整个输入（系统提示 + 工具定义 + 检索结果 + 历史 + 当前输入） |
| **是否动态** | 通常是静态的             | 动态的，每一步都可能不同                         |
| **典型场景** | 单轮问答、文案生成          | Agent 多步循环、RAG 系统、复杂工作流              |
| **操作对象** | Prompt 文本          | 整个上下文窗口的 Token 分配                    |
| **核心问题** | "我该怎么措辞？"          | "模型在这一步应该看到哪些信息？"                    |
| **优化方向** | 更好的指令、更清晰的约束       | 信息选择、排序、压缩、时序编排                      |

### PE 是 CE 的子集

一张图就能说明白：

```text
┌─────────────────────────────────────────────┐
│           Context Engineering               │
│                                             │
│  ┌──────────────────────┐                   │
│  │  Prompt Engineering  │  ← 只是其中一部分  │
│  │  (措辞、指令设计)     │                   │
│  └──────────────────────┘                   │
│                                             │
│  + 信息选择 (RAG、工具定义、记忆)             │
│  + 排序策略 (什么放前面、什么放后面)           │
│  + 压缩策略 (摘要、裁剪、Token 管理)          │
│  + 编排策略 (不同步骤加载不同信息)             │
│                                             │
└─────────────────────────────────────────────┘
```

打个比方：

- **Prompt Engineering** 像是写好一封邮件的措辞——用词精准、语气得体。
- **Context Engineering** 除了写好一封邮件的**措辞**外，还决定这封邮件要**附上哪些附件、引用哪些历史邮件、抄送哪些人**，以及什么时候发。

措辞再好，信息不对，事情也办不成。

## 三、Context Engineering 的四大支柱

Context Engineering 不是一个模糊的概念，它可以拆解为四个具体的能力维度：

### 1. Context Composition（选什么）

> 决定哪些信息进入上下文窗口。

这是最基础的问题。一次 LLM 调用的上下文可能包含：

- System Prompt（角色设定、行为约束）
- 工具定义（Tool Schema）
- RAG 检索结果
- 对话历史
- 用户当前输入
- 外部数据（API 响应、文件内容）

你不可能把所有东西都塞进去。所以你必须做选择：

```python
# 伪代码：Context Composition 的决策过程
def compose_context(user_query, conversation_history, available_tools):
    context = []
    
    # 1. 系统指令 —— 永远在
    context.append(system_prompt)
    
    # 2. 工具定义 —— 根据用户意图选择性加载
    relevant_tools = select_tools(user_query, available_tools)
    context.append(format_tools(relevant_tools))
    
    # 3. 检索结果 —— 只在需要外部知识时加载
    if needs_retrieval(user_query):
        docs = retrieve(user_query, top_k=3)
        context.append(format_docs(docs))
    
    # 4. 对话历史 —— 不是全部，而是最近 N 轮或已压缩
    recent_history = conversation_history[-10:]
    context.append(format_history(recent_history))
    
    # 5. 用户当前输入
    context.append(user_query)
    return context
```

**核心原则：只加载当前步骤需要的信息，而不是"反正窗口够大，全塞进去"。**

### 2. Context Ranking（排什么序）

> 信息的排列顺序影响模型的注意力分配。

大量研究（包括著名的"**Lost in the Middle**"论文）表明，LLM 对上下文**开头和结尾**的信息关注度最高，中间部分容易被忽略。所以信息的排列顺序非常重要：

```text
高关注 ──→ System Prompt（角色、核心约束）
高关注 ──→ 当前最相关的工具定义
         ──→ 检索到的参考文档
         ──→ 较早的对话历史        ← 容易被忽略的"中间地带"
高关注 ──→ 最近几轮对话
高关注 ──→ 用户当前输入
```

**实践技巧：把最重要的信息放在开头和结尾，中间放相对不那么关键的内容。**

### 3. Context Optimization（怎么压缩）

> 在有限的上下文窗口里，最大化信号密度。

上下文窗口虽然越来越大（128k、200k、甚至 1M），但更大不等于更好。塞太多信息会导致：

- **信号被噪声稀释**：模型在 100k Token 的"大海"里找一条关键信息，效果远不如在 5k Token 的"池塘"里找
- **成本飙升**：Token 是按量计费的
- **延迟增加**：输入越长，首 Token 时间（TTFT）越慢

**核心认知：Context Optimization 不是"把上下文压得越短越好"，而是"把最有用的信息，以最便宜、最稳定、最容易被模型利用的形式送进去"。**

#### 六大压缩策略一览

| 策略               | 核心思路                | 典型做法                                        | 最佳适用场景        |
| ---------------- | ------------------- | ------------------------------------------- | ------------- |
| **① 相关性裁剪**      | 先过滤无关内容，再压缩         | 识别用户意图 → 删除无关消息/日志/代码 → 只对候选内容压缩            | 多轮 Agent、长对话  |
| **② RAG + 压缩**   | 先检索，再压命中片段          | top-k 召回 → 去重/合并 → query-focused 抽取或摘要      | 知识库问答、文档助手    |
| **③ 分层记忆**       | 短期原文 + 中期摘要 + 长期结构化 | Layer1 当前轮原文 / Layer2 滚动摘要 / Layer3 JSON 状态 | Agent 工作流、长会话 |
| **④ 上下文缓存**      | 稳定前缀不重传             | system prompt、tool schema、few-shot 示例做前缀缓存  | 重复分析同一文档/代码库  |
| **⑤ 上下文化 Chunk** | 给每个 chunk 补语义定位     | 生成 50-100 tokens 的上下文前缀（文档主题+章节+讨论对象）       | 法务/政策/技术规范文档  |
| **⑥ 按问题压缩**      | 围绕当前 query 做定向压缩    | 只保留支持回答的事实、数字、条件；删背景和修辞                     | QA、高风险场景      |

一句话理解裁剪优先级：**不是"把 100 页变 10 页"，而是"先把不该看的 92 页扔掉，再把剩下 8 页压成 2 页"。**

#### 四类压缩方式对比

| 压缩方式 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| **抽取式**（摘关键句） | 保真度高，易溯源 | 不够短，可读性一般 | 法务、金融、代码错误定位 |
| **生成式**（模型重写摘要） | 压缩率高，可结构化 | 可能漏/错事实 | 聊天历史、会议纪要 |
| **结构化**（JSON/表格/槽位） | 最稳定，易 diff/merge/校验 | schema 设计有成本 | Agent state、任务编排 |
| **语义检索替代**（存外部，命中再取） | 全量知识可保留 | 依赖检索质量 | 知识库、企业搜索 |

#### 分层记忆的结构化示例

摘要不要写成散文，写成结构化槽位，模型最容易利用：

```json
{
  "goal": "当前任务目标",
  "constraints": ["约束条件"],
  "decisions": ["已做决策"],
  "open_questions": ["未解决问题"],
  "artifacts": ["关键产物引用"],
  "next_steps": ["下一步行动"]
}
```

#### 实战建议

1. **先裁剪，再压缩**——相关性过滤是收益最大的第一步
2. **结构化 > 文学性摘要**——表格、JSON、决策记录比一段漂亮 prose 更稳
3. **压缩要可评估**——答案准确率、引用正确率、关键事实召回率，没有评估的"优化"只是自我感动
4. **缓存 ≠ 压缩**——缓存解决重复传输成本，压缩解决窗口容量和注意力分配，两者常组合使用

**核心目标：每个 Token 都要"值得"占据上下文窗口的位置。**

### 4. Context Orchestration（何时加载）

> 不同步骤加载不同的信息。

这是 Agent 场景独有的挑战。一个 Agent 在执行任务时可能经历多个步骤，每一步需要的信息完全不同：

```text
步骤 1（理解用户意图）:
  → 需要：对话历史 + 用户输入
  → 不需要：工具定义、检索结果

步骤 2（规划行动）:
  → 需要：可用工具列表 + 任务分解指令
  → 不需要：完整对话历史

步骤 3（执行搜索）:
  → 需要：搜索工具定义 + 用户查询
  → 不需要：其他工具定义

步骤 4（生成最终回答）:
  → 需要：搜索结果 + 用户原始问题 + 输出格式要求
  → 不需要：中间步骤的推理过程
```

**关键思维转变：不要把上下文当作一个"静态配置"，要把它当作一个"动态节目单"——每一幕上演不同的场景，需要不同的道具和演员。**

## 五、LangChain 的四种策略：Write / Select / Compress / Isolate

LangChain 团队在总结 Context Engineering 的实践经验时，提炼出了四种核心策略。这四种策略是对四大支柱的另一种视角，更侧重"怎么做"：![CleanShot%202026-04-13%20at%2014.59.00.png](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/CleanShot%202026-04-13%20at%2014.59.00.png)

### 1. Write（写出去）

> 把信息写到外部存储，需要时再取回来。

Write上下文指的是将其保存在上下文窗口之外，以帮助代理执行任务。 可以通过将其写入以下位置来实现： 
- 长期记忆（跨会话持久化） 
- 短期记忆（会话内持久化） 
- 状态对象
![](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/20260413150131196.png)

模型上下文窗口是有限的"短期记忆"。如果需要跨会话、跨步骤保留信息，就得写到外部。

### 2. Select/Read（选进来）

> 从外部拉取当前步骤需要的信息。

读上下文意味着将其拉入上下文窗口，这个上下文可以从以下来源拉取： 
- 工具 
- 记忆 
- 知识库（文档，向量数据库）
![](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/20260413150233286.png)

> 用更少的 Token 表达同样的信息。

### 4. Isolate（隔离）

> 把不同的信息放在不同的"空间"里，互不干扰。

隔离上下文涉及将其拆分，以帮助代理执行任务。 一些常用的方法包括： 
- 使用多个代理（或子代理），每个都有自己的上下文 
- 使用沙箱环境进行代码存储和执行

**隔离的好处：每个 Agent 只看到自己需要的信息，上下文干净、聚焦、Token 高效。**

## 六、上下文 = 全部输入信息

为了让你对"上下文到底包含什么"有一个直观认识，我们来解剖一次典型的 LLM API 调用：

```text
┌─────────────────────────────────────────────────────┐
│              一次 LLM 调用的上下文构成                  │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ① System Prompt（系统指令）           ~200 tokens   │
│     ├── 角色设定："你是一位电商客服..."                │
│     ├── 行为约束："不要编造信息..."                    │
│     └── 输出格式："回答控制在100字以内..."             │
│                                                     │
│  ② 工具定义（Tool Definitions）       ~800 tokens   │
│     ├── search_products: {...}                      │
│     ├── check_order: {...}                          │
│     └── create_refund: {...}                        │
│                                                     │
│  ③ 检索结果（RAG Results）            ~1500 tokens   │
│     ├── 退货政策文档片段                              │
│     └── 用户历史订单信息                              │
│                                                     │
│  ④ 对话历史（Conversation History）   ~3000 tokens   │
│     ├── [旧对话摘要]                                 │
│     ├── user: "我想退货"                             │
│     ├── assistant: "好的，请提供订单号"                │
│     ├── user: "DD20250612001"                       │
│     ├── assistant: [调用 check_order]                │
│     ├── tool: {订单详情...}                          │
│     └── assistant: "您的订单状态是..."                │
│                                                     │
│  ⑤ 用户当前输入                        ~30 tokens    │
│     └── user: "退货单号是多少？"                      │
│                                                     │
├─────────────────────────────────────────────────────┤
│  总计: ~5530 tokens                                  │
│  每个部分都消耗 Token，都需要权衡                       │
└─────────────────────────────────────────────────────┘
```

看到了吗？用户只说了一句"退货单号是多少"（30 tokens），但模型实际看到的是 **5530 tokens** 的上下文。

**Context Engineering 就是设计和管理这 5530 个 Token 的学问。**

- **①② 相对稳定**：系统提示和工具定义在一个会话中基本不变（但可以跨步骤动态调整）
- **③ 按需加载**：只在用户提到退货时才检索退货政策
- **④ 持续增长**：这是最需要管理的部分——不压缩会不断膨胀
- **⑤ 每轮更新**：用户的新输入

每个部分都有优化空间，每个选择都影响模型的表现。这就是为什么 Cognition 说 Context Engineering 是"头号工作"。

## 小结

回顾一下这篇的核心要点：

1. **Prompt Engineering 解决"怎么问"，Context Engineering 解决"模型看到什么"**——当你从单轮对话走向 Agent，后者才是真正的瓶颈。
2. **Karpathy 的类比**：LLM 是 CPU，上下文窗口是 RAM，你是操作系统。你的工作是**内存管理**，不是写措辞。
3. **四大支柱**：
   - **Composition**（选什么）：只加载当前步骤需要的信息
   - **Ranking**（排什么序）：重要信息放开头和结尾
   - **Optimization**（怎么压缩）：最大化每个 Token 的信号密度
   - **Orchestration**（何时加载）：不同步骤、不同上下文
1. **LangChain 四策略**：Write（写出去）、Select（选进来）、Compress（窗口内压缩）、Isolate（多空间隔离）
2. **PE 是 CE 的子集**：Prompt Engineering 是 Context Engineering 的一部分，但远不是全部。

> 💡 **预告**：理解了 Context Engineering 的定义和框架，下一篇我们要回答一个更关键的问题——**为什么 Agent 特别需要 Context Engineering？** 我们会深入"Lost in the Middle"问题、Context Rot（上下文腐烂）现象，以及 Agent 多步循环中每一步都需要不同最优上下文的挑战。当你理解了"为什么"，才能真正把这些策略用到自己的 Agent 里。
