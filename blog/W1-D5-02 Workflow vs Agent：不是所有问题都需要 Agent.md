# Workflow vs Agent：不是所有问题都需要 Agent

---

> 摘要：学了几天 Agent 概念，是不是已经跃跃欲试想给所有项目都加上 Agent 了？先别急。Anthropic 在他们的 Building Effective Agents 中给出了一个可能让你意外的建议："大部分场景用 Workflow 就够了"。这篇我们来聊聊 Workflow 和 Agent 的本质区别、Anthropic 定义的 5 种 Workflow 模式，以及一个关键问题——你的任务到底该用哪个？

## 一、Anthropic 的定义：Workflow vs Agent

Anthropic 在 Building Effective Agents 中把 LLM 应用分成了两大类：

- **Workflows（工作流）**: 系统通过LLM和工具进行编排 
- **Agents（智能体）**: LLM 动态指导自身流程和工具使用的系统

一句话说清区别：**Workflow 是"你告诉它怎么走"，Agent 是"它自己决定怎么走"。**

| 维度 | Workflow | Agent |
|------|----------|-------|
| 控制流 | 预定义的代码路径 | 模型动态决策 |
| 可预测性 | 高 | 低 |
| 灵活性 | 低 | 高 |
| 调试难度 | 低 | 高 |
| 成本 | 较低（LLM 调用次数可控） | 较高（循环次数不确定） |
| 延迟 | 可预测 | 不可预测 |
| 适用场景 | 结构化、定义明确的任务 | 开放式、需要灵活决策的任务 |

看到这个表，你可能会想：Agent 的灵活性更高，那不是更强吗？没错，但"更强"不等于"更好"。**灵活性是有代价的——你用不到的灵活性，就是纯粹的负担。**

## 二、5 种 Workflow 模式

在你急着搞 Agent 之前，先看看 Anthropic 总结的 5 种 Workflow 模式。很多你以为需要 Agent 的任务，其实用这些模式就能搞定。

![](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/ChatGPT%20Image%202026%E5%B9%B44%E6%9C%8817%E6%97%A5%2017_49_25.png)

### 模式 1：Prompt Chaining（提示链）

```text
┌──────────┐     ┌──────┐     ┌──────────┐     ┌──────────┐
│  Task A  │ ──→ │ Gate │ ──→ │  Task B  │ ──→ │  Task C  │
│ (LLM 1)  │     │ 质检  │     │ (LLM 2)  │     │ (LLM 3)  │
└──────────┘     └──────┘     └──────────┘     └──────────┘
```

**什么时候用？** 任务能拆成固定的几个步骤，前一步的输出是后一步的输入。

比如翻译例子——先翻译、再润色，就是最经典的 Prompt Chaining。中间还能加个 Gate（质检），如果翻译质量不达标就重做，达标了才进入润色环节。

### 模式 2：Routing（路由）

```text
                      ┌──────────────────┐
                  ┌─→ │ 处理路径 A（技术）  │
┌──────────┐     │   └──────────────────┘
│  分类器   │ ──→─┤
│ (Router)  │     │   ┌──────────────────┐
└──────────┘     ├─→ │ 处理路径 B（商务）  │
                  │   └──────────────────┘
                  │   ┌──────────────────┐
                  └─→ │ 处理路径 C（日常）  │
                      └──────────────────┘
```

**什么时候用？** 输入有明确的类别，不同类别需要不同的处理逻辑。

典型场景：客服系统。用户的问题先过一个分类器——技术问题走技术知识库，退款问题走退款流程，闲聊走通用对话。每条路径都是确定的，不需要 Agent 来"思考"该走哪条路。

### 模式 3：Parallelization（并行化）

```text
方式一：Sectioning（分段）        方式二：Voting（投票）

  ┌──────────┐                    ┌──────────┐
  │ 子任务 A  │                    │  LLM 1   │
  └────┬─────┘                    └────┬─────┘
       │                               │
  ┌──────────┐      汇总          ┌──────────┐      投票
  │ 子任务 B  │ ───→ 合并          │  LLM 2   │ ───→ 聚合
  └────┬─────┘                    └────┬─────┘
       │                               │
  ┌──────────┐                    ┌──────────┐
  │ 子任务 C  │                    │  LLM 3   │
  └──────────┘                    └──────────┘
```

**什么时候用？** 任务可以拆成互不依赖的子任务同时执行，或者需要多个模型"投票"提高可靠性。

- **Sectioning（分段）**：代码审查时，同时检查安全性、代码风格、性能，最后汇总。
- **Voting（投票）**：内容审核时，让 3 个 LLM 分别判断是否违规，多数票决定结果。

### 模式 4：Orchestrator-Workers（编排）

```text
                ┌────────────────┐
                │   Orchestrator │
                │   (编排器 LLM)  │
                └───┬───┬───┬───┘
                    │   │   │
              ┌─────┘   │   └─────┐
              ↓         ↓         ↓
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Worker A │ │ Worker B │ │ Worker C │
        │  (LLM)   │ │  (LLM)   │ │  (LLM)   │
        └──────────┘ └──────────┘ └──────────┘
              │         │         │
              └─────┐   │   ┌─────┘
                    ↓   ↓   ↓
               ┌────────────────┐
               │    合并结果      │
               └────────────────┘
```

**什么时候用？** 子任务的数量和内容无法提前确定，需要一个"指挥官"来动态分配。

比如"把这个代码仓库迁移到新框架"——编排器分析仓库后决定哪些文件需要改、怎么改，然后分配给不同的 Worker 去执行。注意，虽然编排器是 LLM 在做决策，但**整体流程是预定义的**（分析 → 分配 → 执行 → 合并），所以它还是 Workflow，不是 Agent。

### 模式 5：Evaluator-Optimizer（评估-优化）

```text
┌──────────────────────────────────────────┐
│                                          │
│   ┌──────────┐      ┌──────────────┐     │
│   │ Generator │ ──→ │  Evaluator   │     │
│   │  (生成器)  │     │   (评估器)    │     │
│   └──────────┘      └──────┬───────┘     │
│        ↑                    │             │
│        │    不合格，附带反馈   │             │
│        └────────────────────┘             │
│                                          │
│        合格 → 输出最终结果                  │
└──────────────────────────────────────────┘
```

**什么时候用？** 有明确的评估标准，可以通过"生成 → 评估 → 改进"的循环来提升质量。
如文学翻译——先翻译一版，评估器检查信达雅，不达标就带着反馈让生成器改进。

## 四、真正的 Agent 长什么样？

![](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/20260417180831662.png)

> Agent 可以处理复杂任务，但它们的实现通常很简洁。本质上就是 LLM 在循环中根据环境反馈使用工具。


**区分 Agent 和 Workflow 的关键特征：**

1. **循环次数不确定**——不是"固定 3 步"，而是"做到完成为止"，可能 2 步，也可能 20 步
2. **模型自己决定下一步**——不是代码写死"先做 A 再做 B"，而是模型根据当前状态决定
3. **需要从环境获取反馈**——执行一个操作后，观察真实结果（比如运行代码看报错），再决定下一步
4. **可以在检查点暂停**——等待人类确认后再继续（Human-in-the-Loop）

**典型的 Agent 场景：**

- Coding Agent（如 Claude Code、Cursor）：给一个 bug 描述，Agent 自己读代码、定位问题、修复、跑测试、发现新问题、继续修……直到测试全过
- 浏览器 Agent：给一个研究任务，Agent 自己搜索、打开页面、提取信息、发现新线索、继续搜……直到信息足够

## 五、判断标准——我的任务该用 Workflow 还是 Agent？

说了这么多概念，最实用的是这棵决策树：

![](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/20260417181129804.png)

再来几个具体例子帮你建立直觉：

| 任务 | 推荐方案 | 为什么 |
|------|---------|--------|
| 翻译一篇文章 | Prompt Chaining | 固定步骤：翻译 → 润色 → 输出 |
| 客服问题分类和处理 | Routing | 明确类别，每类有固定处理流程 |
| 代码审查（安全+风格+性能） | Parallelization | 多个独立维度同时检查 |
| 根据用户需求重构代码仓库 | Orchestrator-Workers | 子任务数量不确定，但流程确定 |
| 修复一个复杂的 bug | Agent | 步骤不可预测，需要不断试错 |
| 自主完成编程任务（SWE-bench） | Agent | 开放式任务，需要灵活决策 |

**Q：一个 Chat 应该用 Workflow 还是 Agent？**

看两点：**流程是否固定**、**是否依赖环境反馈**。  

- **固定流程 + 固定工具规则** → Workflow  
  比如客服 Chat：先意图识别，再走不同处理路径（Routing），必要时加质检（Evaluator）。
- **步骤不确定 + 需要反复试错** → Agent  
  比如调试类 Chat：读日志、改代码、跑测试、看报错，再决定下一步。

**Q：有一堆工具调用怎么选？**

- **工具清单是固定的，调用顺序也能画出来** → Workflow（Chaining + Routing）
- **工具清单不固定，需要边做边决定** → Agent（模型在循环中自己选工具）

**一个简单的经验法则：如果你能画出完整的流程图，用 Workflow；如果流程图里出现了"不确定"和"看情况"，考虑 Agent。**

## 六、过度设计的代价

最后聊聊为什么要克制使用 Agent 的冲动。过度设计的代价是实实在在的：

**1. 更贵**——Agent 的 LLM 调用次数不可控。一个简单任务，Workflow 可能 2 次调用搞定，Agent 可能绕了 8 次。每次调用都是真金白银。

**2. 更慢**——每次 LLM 调用都有延迟。Agent 的串行循环意味着延迟是累加的，而且你无法预测要累加多少次。

**3. 更难调试**——Workflow 每次走的路径是一样的，出了问题很容易定位。Agent 每次走的路径可能不同，今天能跑通不代表明天也行。非确定性路径是 debug 的噩梦。

**4. 错误会累积**——Agent 循环中，每一步都可能出错，而且前一步的错误会影响后续所有步骤。5 步循环中每步 95% 正确率，最终正确率只有 95%^5 ≈ 77%。10 步呢？60%。这就是 **compounding errors（复合错误）** 的威力。

所以，**不是 Agent 不好，而是要用在对的地方。** 就像你不会用挖掘机来种花——能行，但何必呢？

## 小结

1. **Workflow 是预定义路径，Agent 是动态决策。** 前者是"你告诉它怎么走"，后者是"它自己决定怎么走"。
2. **大多数场景用 Workflow 就够了。** Anthropic 总结了 5 种 Workflow 模式——Prompt Chaining、Routing、Parallelization、Orchestrator-Workers、Evaluator-Optimizer——覆盖了绝大多数结构化任务。
3. **只在任务需要灵活决策、步骤不可预测时才用 Agent。** 典型场景是复杂 bug 修复、自主编程、开放式研究。
4. **过度设计 = 更贵 + 更慢 + 更难调试 + 错误累积。** 找到最简单的可行方案，才是真正的工程智慧。
