# 什么是 AI Agent：不只是能聊天，而是能"做事"的系统

---

> 摘要：所有人都在说 Agent，但它到底是什么？我一开始以为 ChatGPT 本身就是 Agent，后来发现这个理解大错特错。这篇从 Anthropic 官方定义出发，搞清楚 Agent 和 Chatbot 的本质区别，拆解 Agent 的五大核心特征，深入理解它的核心引擎——感知-推理-行动循环，并用完整示例和代码走通一个最小的 Agent 循环实现。理解了这些，你才知道后面要学的东西为什么是这个顺序。

## 一、Anthropic 怎么定义 Agent

Anthropic 在他们的博客里给了一个非常清晰的定义：

> Agent 是 LLM 能**自主决定自己的流程和工具使用**的系统，全程掌控任务的完成方式。

也就是：

1. 它自己决定下一步做什么
2. 它自己选择用什么工具
3. 它自己判断做完了没有
4. 做完了才停下来

这和我们平时用 ChatGPT 的体验完全不一样。

---

## 二、Agent ≠ Chatbot：到底区别在哪

我做了一张对比表，帮你理清思路：

| 维度         | Chatbot（聊天机器人） | Agent（智能代理）     |
| ---------- | -------------- | --------------- |
| **交互模式**   | 一问一答，你问它答      | 接受目标，自主执行多步任务   |
| **输入/输出**  | 文本进 → 文本出      | 目标进 → 行动 + 结果出  |
| **能否调用工具** | 不能（或非常有限）      | 主动调用多种工具        |
| **是否有循环**  | 没有，一次请求一次响应    | 有，感知→推理→行动循环    |
| **是否自主决策** | 不自主，完全被动响应     | 自主决定下一步做什么      |
| **是否有记忆**  | 只有当前对话上下文      | 有工作记忆，可跨步骤追踪状态  |
| **典型场景**   | 闲聊、问答、翻译       | 代码开发、数据分析、自动化流程 |

---

## 三、Agent 的五大核心特征

搞清楚定义后，我试着把 Agent 的关键特征拆解成五个维度：

```text
┌─────────────────────────────────────────────────────┐
│                  AI Agent 五大特征                    │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 自主性（Autonomy）                               │
│     └── 不需要人类逐步指令，自己决定怎么完成任务         │
│                                                     │
│  2. 感知（Perception）                               │
│     └── 能接收环境信息：用户输入、工具返回、文件内容...   │
│                                                     │
│  3. 推理（Reasoning）                                │
│     └── 能分析当前状态，判断下一步该做什么               │
│                                                     │
│  4. 行动（Action）                                   │
│     └── 能调用工具、执行代码、读写文件、发起请求...      │
│                                                     │
│  5. 记忆（Memory）                                   │
│     └── 能记住之前做了什么、结果是什么，避免重复          │
│                                                     │
└─────────────────────────────────────────────────────┘
```

这五个特征缺了任何一个，都不算完整的 Agent：

- 没有**自主性** → 那就是个脚本，每一步都需要人来触发
- 没有**感知** → 瞎子，不知道环境发生了什么
- 没有**推理** → 莽夫，不会根据情况调整策略
- 没有**行动** → 嘴炮，只能说不能做
- 没有**记忆** → 金鱼，做完就忘，反复犯错

## 四、Agent 的心跳：感知-推理-行动循环

![](https://limbo.oss-cn-beijing.aliyuncs.com/wiki/20260416175619332.png)

一个类比：Agent 就像一个实习生，老板丢给你个任务："调研一下竞品的定价策略，整理成表格。" 你会怎么做？

1. **看**——先读一遍任务要求，搞清楚老板要什么（感知）
2. **想**——想想该怎么做：先搜索竞品有哪些，再去各家官网找定价页面（推理）
3. **做**——打开浏览器，开始搜索（行动）
4. **看**——搜索结果出来了，扫一眼有没有有用的信息（感知）
5. **想**——这个结果不太对，换个关键词试试（推理）
6. **做**——换个关键词再搜一次（行动）
7. ……不断循环，直到表格填完

发现没有？你自然而然就在做一个循环：**看 → 想 → 做 → 看 → 想 → 做 → ……**
Agent 干的事情，一模一样。只不过"看"是接收输入，"想"是 LLM 推理，"做"是调用工具。

```text
               ┌──────────────────────────┐
               ▼                          │
        ┌─────────────┐                   │
        │  Perceive   │  ← 接收输入       │
        │  （感知）    │                   │
        └──────┬──────┘                   │
               ▼                          │
        ┌─────────────┐                   │
        │   Reason    │  ← LLM 推理       │
        │  （推理）    │                   │
        └──────┬──────┘                   │
               ▼                          │
        ┌─────────────┐                   │
        │    Act      │  ← 执行动作       │
        │  （行动）    │                   │
        └──────┬──────┘                   │
               │   行动结果变成新的输入     │
               └──────────────────────────┘
```

这就是 Agent 的"心跳"——一次一次地跳，直到任务完成。
一个 Agent 的核心运行逻辑，其实用五行伪代码就能说清楚：

```python
while not done:
    observation = perceive(environment)    # 感知：看到了什么？
    thought = reason(observation, memory)  # 推理：该怎么做？
    action = decide(thought)               # 决策：选择行动
    result = act(action)                   # 行动：执行操作
    memory.update(result)                  # 记忆：记住结果
```

这个循环就是 Agent 的心脏。下面我们把每一步拆开来细看。

---

## 五、感知（Perceive）—— Agent 在"看"什么？

Agent 的感知不是指一双眼睛在看东西，而是指**所有会进入 LLM 上下文窗口的信息**。

具体来说，Agent 会"看"到这些东西：

| 感知来源 | 具体内容 | 什么时候出现 |
|---------|---------|-------------|
| 用户请求 | 用户发过来的原始问题 | 循环的第一轮 |
| 工具返回 | 调用工具后拿到的结果（Observation） | 每次工具执行完 |
| 环境信息 | 文件系统状态、API 响应、报错信息 | 工具执行过程中 |
| 对话历史 | 之前几轮的对话记录 | 每一轮都在 |
| 系统指令 | System Prompt 中的规则和约束 | 每一轮都在 |

这里有个容易忽略的点：**感知不是被动接收，Agent 也可以主动感知**。
这就跟那个实习生一样——老板没给你竞品列表，你得自己去搜。主动搜索的过程本身就是"行动"，搜索结果就是新的"感知"。

在真实 Agent 中，感知阶段对应的是：

```text
┌──────────────┐
│   perceive   │ ← 读取用户输入 / 工具返回结果 / 文件内容 / 错误信息
└──────┬───────┘
       ▼
```

---

## 六、推理（Reason）—— Agent 在"想"什么？

推理是这个循环里最关键的一步，因为它直接决定了 Agent 接下来要干嘛。
LLM 在推理阶段，本质上在回答这几个问题：

```text
┌─────────────────────────────────────────────┐
│  推理阶段的核心问题                           │
│                                             │
│  1. 我现在知道什么？还缺什么？                 │
│     → 分析当前状态                           │
│                                             │
│  2. 下一步该做什么？                          │
│     → 制定计划                              │
│                                             │
│  3. 用哪个工具最合适？参数是什么？              │
│     → 选择工具                              │
│                                             │
│  4. 任务完成了吗？可以给用户回复了吗？           │
│     → 判断终止                              │
└─────────────────────────────────────────────┘
```

来看一个具体的例子，感受一下推理过程：

```text
用户问：北京今天天气怎么样？如果下雨就提醒我带伞。

Agent 推理过程：
Thought 1: 用户想知道北京天气，我需要调用天气API
Thought 2: 还需要根据天气结果判断是否下雨
Thought 3: 如果下雨，要在回复中加上带伞提醒
```

注意 Thought 2 和 Thought 3 不是在第一轮就能完成的——因为 Agent 还没拿到天气数据。它需要先行动（调用天气 API），拿到结果后再感知，然后才能继续推理。

**这里有一个我觉得很重要的认知：推理能力完全来自 LLM 本身。**

> Agent 框架不会让一个弱模型变聪明。模型越强，推理越靠谱；模型太弱，循环再多次也是在瞎转。这也是为什么做 Agent 开发的人普遍建议用最强的模型——推理是整个循环的发动机，发动机不行，车再好也跑不动。

在真实 Agent 中，推理阶段对应的是：

```text
┌──────────────┐
│    reason    │ ← LLM 分析当前状态："文件有 bug，需要修复第 42 行"
└──────┬───────┘
       ▼
┌──────────────┐
│    decide    │ ← 选择行动："调用 edit_file 工具修改代码"
└──────┬───────┘
       ▼
```

---

## 七、行动（Act）—— Agent 在"做"什么？

推理完了，Agent 该动手了。行动主要分三种：

**1. 调用工具（Function Calling）**

这是最常见的行动。比如调用搜索引擎、查天气、读文件、执行代码。

**2. 生成最终回复**

当 Agent 判断任务完成了，它不再调用工具，而是直接输出文本回复给用户。这其实也是一种"行动"——只不过不是对外部环境的操作，而是对用户的输出。

**3. 修改环境**

有些 Agent 会写文件、执行代码、发送消息。这些行动会直接改变外部环境的状态。

这里最关键的一点是：**行动会产生新的感知输入，从而驱动下一轮循环。**

```text
Agent 调用天气 API
     │
     ▼
API 返回：{"city": "北京", "temp": 28, "weather": "晴"}
     │
     ▼
这个结果作为 Observation 进入下一轮的"感知"
     │
     ▼
Agent 基于新的感知继续"推理"
```

这就是循环能跑起来的原因——行动的输出变成感知的输入，形成闭环。

在真实 Agent 中，行动阶段对应的是：

```text
┌──────────────┐
│     act      │ ← 执行工具调用，拿到结果
└──────┬───────┘
       ▼
┌──────────────┐
│    update    │ ← 把结果存储，进入下一轮循环
└──────┬───────┘
       ▼
     done?  ──No──→ 回到 perceive
       │
      Yes
       ▼
    返回最终结果
```

---

## 八、循环什么时候停？

一个关键问题：这个循环总不能永远跑下去吧？当然不能。停下来有几种情况：

**1. 模型判断任务完成了**

这是最正常的终止方式。当 LLM 觉得信息够了、可以回答用户了，它就不再生成工具调用，而是直接输出文本。框架检测到"这一轮没有工具调用"，就知道循环该结束了。

**2. 达到最大迭代次数**

这是一个安全机制，防止 Agent 陷入死循环。实际开发中一定要设这个上限。

**3. 遇到不可恢复的错误**

比如工具执行失败、API 挂了、权限不足。好的 Agent 会尝试重试或换个方案，但如果连续失败，就应该停下来告诉用户。

**4. 用户中断**

用户直接取消了请求。

## 小结

回顾一下今天搞明白的几件事：

1. **Agent 不是"更聪明的 Chatbot"**，它是一个能自主感知环境、做出决策并采取行动的系统。核心区别是"能不能自己去做事"。
2. **Anthropic 的定义**：Agent 是 LLM 能动态决定自己的流程和工具使用的系统——关键在于**自主性**和**控制权**。
3. **五大特征**：自主性、感知、推理、行动、记忆。缺了任何一个都不是完整的 Agent。
4. **Agent 的核心机制就是 Perceive → Reason → Act 循环**，没有什么神秘的，本质上就是一个循环加上 LLM 的判断。
5. **感知是输入**——用户请求、工具返回、环境信息都是感知来源，Agent 也可以主动感知。
6. **推理是大脑**——LLM 分析当前状态、制定计划、选择工具、判断终止，推理能力完全取决于模型本身。
7. **行动是输出**——调用工具、生成回复、修改环境，行动的结果会变成新的感知，驱动循环继续。
8. **循环终止靠两个机制**——模型自己判断"够了"，或者达到最大迭代次数兜底。

一条可带走的经验：**判断一个系统是不是 Agent，就看它能不能在你只给目标的情况下，自己循环执行多步操作直到完成。能循环、能用工具、能自己决定下一步——这就是 Agent。**

如果你理解了这个循环，那你已经抓住了 Agent 最核心的运行逻辑。后面不管学什么框架（LangGraph、CrewAI、OpenAI Agents SDK），底层都是这个循环在跳。

---

> 💡预告 这个循环看起来很美好，但实际项目中你很快会遇到一个问题：是不是所有场景都需要这个循环？有些任务简简单单，让 LLM 按步骤走一遍就行了，根本不需要 Agent 自己判断下一步。强行上循环反而浪费 Token、增加延迟。下一篇我们要聊 Anthropic 在 *Building Effective Agents* 中提出的一个重要区分：**Workflow vs Agent。什么时候用固定流程编排，什么时候让 Agent 自己跑循环——搞清楚这个问题，才能避免"为了 Agent 而 Agent"的陷阱。
